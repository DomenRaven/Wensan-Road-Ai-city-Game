## 一、项目背景与场地条件

### 1.1 背景

文三路数字生活街区是全国首个数字生活街区样板，本馆定位为 **「文三·中国 AI 新源地 / AI 杭州·第一城市客厅」**，承接街区「从买电脑到玩未来」的叙事升级。全馆 V2.0 策略从「AI 垂直超市陈列」转向 **「解决问题的生活客厅」**，以四问（学 / 生 / 办 / 健）组织动线。

**AI 教育区**原以 K12 设备陈列为主；升级后需在 **SamThree 问题导向学习** 旁侧，增设可体验的 **「AI 辅助游戏创作」** 展项，让 6–15 岁学生及亲子家庭在约 **10 分钟内**完成「提需求 → AI 生成 → 试玩验收」闭环，直观感受 **「AI 是创作伙伴，不是代写作业的工具」**。

本模块与以下馆区叙事衔接：

| 关联板块 | 衔接方式 |
|----------|----------|
| AI 学习 · SamThree | 同一「AI 教育」动线；学习区回答「我学得会吗」，本区回答「我能创造吗」 |
| AI 体育 / 赛车 | 体育竞速类小游戏模板可互引 |
| 路演会议厅 | 试玩片段、创作过程屏录纳入全馆无感成片素材 |
| 序厅机械屏 | 地缘叙事「文三街 · 中国 AI 新源地」品牌一致 |

## 二、产品定位

| 维度 | 说明 |
|------|------|
| 项目名称 | **AI 小游戏创作工坊**（英文名 GameForge K12） |
| 展陈位置 | AI 教育区 · K12 版块（与 SamThree 学习区同动线） |
| 核心价值 | 展厅 **11 类单机小游戏** 联网即玩；**≤10 人同时使用**；优先 preset，逐步 AI 换皮 |
| 用户身份 | 「**小小游戏设计师**」— 选品类 → 试玩（D10）；L1 起可见 AI 改配置 |
| 体验时长 | 单局 **2–3 分钟**；Session 含排队 **≤8 分钟** |
| 适龄 | **14 岁以下优先**；无暴力血腥；多数品类 **≤3 键**（格斗/赛车备案 ≤4 键） |
| 游戏品类（暂定 11） | 射击(FPS/TPS) · 横版格斗 · 割草肉鸽 · 塔防 · 跑酷 · 横版闯关 · 生活模拟 · 体育竞速 · 乒乓球 · 赛车 · 雷霆战机 |
| 演示方式 | **Windows .exe 单机** 经服务器分发/启动；终端 **浏览器/Kiosk 联网** 选游戏 |
| 部署 | **展陈服务器**（OS/云/物理机 **待定**）+ 最多 **10 终端** |
| 与整馆联动 | 创作过程屏录 + 试玩高光 → 路演厅全馆成片素材层 |

**不要做成**：纯 HTML 网页小游戏展台、Steam 级 3A 工坊、无护栏的开放 ChatGPT 聊天窗。

**要做成**：**「AI 创作透明厨房」** — 观众看得见模板、看得见改代码、看得见跑游戏。

## 三、系统架构

采用 **「展陈服务器 + 单机游戏实例 + 终端联网访问」** 架构。**v1.2 取消「展陈后置」**：D1 起落地编排层。

| 层级 | 职责与组件 | v1.2 |
|------|-----------|------|
| **终端层** | 浏览器 / Kiosk · 选品类 · 排队 · 观战 | **D6 起** |
| **编排层** | FastAPI · Redis（**Session≤10**）· nginx · MinIO | **D1 起必做** |
| **制作层** | Godot 4.6 · 模板 · godot-mcp · Cursor · RAG | 持续 |
| **游戏层** | **单机 exe/进程** · `workspace/{id}` 隔离 · demo_preset | **D10：11 类 L0** |
| **监控层** | 健康检查 · Session 日志 · 讲解员复位 API | D7 起 |

> 详图：[`架构/系统架构说明_v1.0.md`](./架构/系统架构说明_v1.0.md) · 日历：[`AI生成小游戏_十天上线路线_v1.2.md`](./AI生成小游戏_十天上线路线_v1.2.md)

### 3.1 模块通信架构

| 通信路径 | 技术实现 | 延迟目标 |
|----------|----------|----------|
| Kiosk → 编排层 | REST + WebSocket | <100ms |
| 编排层 → Cursor Agent | Cursor SDK / CLI 触发 | 会话级 |
| Agent → Godot | godot-mcp（run_project / get_debug_output） | 运行闭环 <3min |
| 编排层 → 展示屏 | Socket.IO 广播进度 | P99 <200ms |
| 生成完成 → 试玩 | 本地 .exe 或 Godot `--path` 启动 | <5s 启动 |
| 产物 → 观众 | MinIO 签名 URL + QR | 24h 有效 |

## 四、五大体验模块详细设计

### 4.1 模块一：智能需求采集（Kiosk 交互）

| 项目 | 内容 |
|------|------|
| 核心功能 | 引导观众用语音/文字/点选完成「品类 + 主题 + 难度」三要素采集 |
| 硬件 | 55″ 立式触摸一体机（竖屏） |
| 交互流程 | 待机轮播 → 选品类（7 类图标）→ 选主题（太空/森林/校园…）→ 选难度（简单/标准）→ 确认生成 |
| 学段适配 | 小学：大图标 + 语音引导；初中+：可文字细调参数 |
| 内容安全 | 敏感词过滤；禁止生成暴力/恐怖/赌博类描述 |
| 时长 | ≤2 分钟 |

### 4.2 模块二：AI 透明创作（过程可视化）

| 项目 | 内容 |
|------|------|
| 核心功能 | 49″ 横屏实时展示 AI 创作过程：选模板 → 改脚本 → 运行 → 修错 |
| 技术 | Cursor Agent + godot-mcp；进度通过 Socket 推送 |
| 可视化阶段 | ① 匹配模板 ② 生成代码 ③ 运行调试 ④ 修复报错 ⑤ 导出完成 |
| 教育价值 | 非黑盒：显示「AI 正在修改 player.gd」等可读步骤 |
| 护栏 | 最多自动修复 **2 轮**；超时回退预置 Demo |
| 时长 | 4–5 分钟 |

### 4.3 模块三：品类模板引擎（核心数据预制 + 参数化小改）

> **设计理念**：每个品类的**移动方式、命中逻辑、数值公式、交互手感**在 `core/` 层人工预制完成；用户创作时 AI **只改 `config/game_config.json` 的 tuning/theme**，不做系统级重写。详见 [`模板引擎/品类核心参数规格_v1.0.md`](./模板引擎/品类核心参数规格_v1.0.md)。

| 品类 | core 层预制（锁定） | tuning 层（AI 小改） | theme 层（换皮） |
|------|---------------------|----------------------|------------------|
| 射击 | 八向移动平滑、AABB 命中、波次刷怪 | 速度、射速、敌人数值 | 太空/森林贴图 |
| 塔防 | 网格规则、伤害公式、波次调度 | 塔伤害、经济、波次 | 地图与塔皮 |
| 体育竞速 | 自动跑、跳跃缓冲、障碍生成 | 速度、障碍密度 | 赛道主题 |
| 横版闯关 | 加速摩擦、coyote、可变高度跳 | 移速、跳跃力、关卡 | 角色与地块 |
| 生活模拟 | 点击热区、线性任务链 | 任务文案 | 房间物品 |
| 推理解谜 | 物品组合规则表 | 谜题表、线索 | 场景皮肤 |
| 休闲 | 子玩法逻辑、计分公式 | 下落速度、刷怪率 | 美术包 |

**创作耗时分布**：复制模板 10s → 改 config 1–2min → 换皮 1min → 运行验证 1min（其余为 AI 编排与展示）

**原则**：**core 预制 + tuning 小改 + theme 换皮**；禁止单次会话新建引擎级系统或修改 `core/` 脚本。

### 4.4 模块四：试玩验收

| 项目 | 内容 |
|------|------|
| 核心功能 | 生成完成后全屏试玩；支持键鼠（WASD/空格） |
| 演示方式 | P0：Godot 直接运行；P1：Windows `.exe` 导出 |
| 操作上限 | ≤3 个按键；单局 2–3 分钟 |
| 降级 | 导出失败 → Godot 窗口运行；仍失败 → 播放预录 Demo 视频 |
| 时长 | 2–3 分钟 |

### 4.5 模块五：成果交付与上墙

| 项目 | 内容 |
|------|------|
| 核心功能 | 扫码下载 `.exe` 或游玩说明卡；优秀作品投屏上墙 |
| 存储 | MinIO 本地对象存储 |
| 隐私 | 默认匿名 Session；可选留昵称上墙（讲解员审核） |
| 全馆联动 | 屏录片段写入路演厅素材池 |
| 时长 | ≤1 分钟 |

## 五、技术选型论证

### 5.1 游戏引擎选型

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Godot 4.x + GDScript** ★ | 全文本项目 AI 友好；2D 小游戏极快；原生导出；godot-mcp 成熟 | 3A 级不适合 | **主选** |
| Unity + C# | 生态大 | 授权成本；AI 改场景重；展陈 PC 负担大 | 不选 |
| **HTML5 原生**（Phaser 等） | 浏览器即玩 | 用户明确不喜欢；性能与体验弱 | **仅降级** |
| Unreal | 画面好 | 过重；不适合 10 分钟小游戏 | 不选 |

**锁定**：Godot **4.6.3 Standard**（非 .NET）+ **GDScript 强类型** + 安装路径 `F:\Godot\Godot_v4.6.3-stable_win64.exe\`

### 5.2 AI 编码选型

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Cursor Agent + godot-mcp** ★ | 多文件编辑；运行/debug 闭环；已验证 4.6.3 | .tscn 手改易错；需 Rules 限 scope | **主选** |
| **本地 RAG（SQLite FTS5）** ★ | Survey-01/02 + 文档可检索（**1348 chunks**）；降幻觉 | 需重建索引；非语义向量 | **主选（P1）** |
| 纯 ChatGPT 网页 | 门槛低 | 无运行闭环 | 不选 |
| godot_ai_assistant 插件 | 引擎内 Agent | 与 Cursor 割裂；展陈 PC 多进程复杂 | **三期参考** |

**Agent 强制流程**：`query_rag.py` → 只改 `config/game_config.json` → `run_project` → 最多 2 轮修复。

### 5.3 前端 / 交互选型（v1.1 下调优先级）

| 阶段 | 方案 | 结论 |
|------|------|------|
| **P1 本地开发** | Cursor 对话 + Godot 编辑器/运行窗口 | **默认** |
| **P2 展陈过渡** | Godot 内 `menu.tscn` 选品类/主题 | 操作简单，与 MCP 同栈 |
| **P3 展陈正式** | React Kiosk + 过程大屏 | 有运营需求再做 |

### 5.4 后端编排选型

| 阶段 | 方案 | 结论 |
|------|------|------|
| P1 | 文件夹 `workspace/{id}` + 批处理脚本 | **够用** |
| P3 | FastAPI + Redis + Socket.IO | 双工位展陈时再上 |

### 5.5 导出与演示优先级

| 优先级 | 方式 | 场景 |
|--------|------|------|
| P0 | Godot MCP `run_project` | 开发调试、展陈日常 |
| P1 | Windows `.exe` 导出 | 观众扫码带走 |
| P2 | Android APK | 二期研学礼包 |
| P3 | HTML5 Web | 无法安装时的降级 |

## 九、风险与降级

| 风险 | 概率 | 降级策略 |
