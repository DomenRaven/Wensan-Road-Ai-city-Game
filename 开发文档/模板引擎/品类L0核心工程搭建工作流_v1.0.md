# 品类 L0 核心工程搭建工作流 v1.0

> **用途**：剩余 10 个品类 Godot 模板的标准作业流程（Agent + 人工审理）  
> **读者**：Cursor Agent · 项目负责人 · 讲解员技术对接  
> **生效**：**审理通过后**按本流程逐品类执行；**单次会话只做一个** `templates/{genre}/`  
> **参考样板**：`templates/platformer/`（D2 已完成 · MCP 无 ERROR）  
> **对齐**：十天上线路线 v1.2 · 品类核心参数规格 v1.0 · `.cursor/rules/godot-mini-game.mdc`

---

## 一、总览：一个大循环 + 一个内循环

```text
┌─────────────────────────────────────────────────────────────────┐
│  单品类完整流程（通过后再进下一品类）                              │
├─────────────────────────────────────────────────────────────────┤
│  ① 阅读要求                                                      │
│  ② 爬取资料                                                      │
│  ③ 细化完善要求 · 更新文档                                        │
│  ④ 正式工程搭建                                                  │
│  ⑤ Godot 测试与打分                                              │
│  ⑥ 优化调整 ←──┐  （小循环：改 → 再测 → 再打分，最多 2 轮）      │
│       ↑        │                                                  │
│       └────────┘  总分 < 80 或硬门槛未过                          │
│  ⑦ 打分评审通过（人工审理 ≥80 且无硬门槛 FAIL）                    │
│  ⑧ 记录当前状态 · 更新文档                                       │
│  ⑨ 下一个品类创作 ──→ 回到 ①                                     │
└─────────────────────────────────────────────────────────────────┘
```

**硬门槛（任一 FAIL = 不得进入 ⑦，必须在 ⑥ 修复）**

| # | 门槛 | 判定 |
|---|------|------|
| H1 | MCP `run_project` | `errors` 数组为空（无 ERROR） |
| H2 | 目录边界 | 本会话 **仅** 修改 `templates/{genre}/` |
| H3 | core 锁定 | 玩法在 `core/`；AI 将来只改 `config/game_config.json` |
| H4 | K12 安全 | 无暴力血腥恐怖；操作 ≤3 键（备案 4 键需注明） |
| H5 | 素材来源 | 仅用 `assets/` · Kenney CC0 |
| H6 | 场景约束 | 场景深度 ≤5 层；单场景节点 ≤80 |

## 二、逐步说明（①–⑨）

### ① 阅读要求（约 15–30 分钟）

**目标**：搞清「这种游戏要做什么、不能做什么」，不急着写代码。

**必读清单**

| 顺序 | 文档 / 配置 | 关注什么 |
|------|-------------|----------|
| 1 | `config/genre_registry.json` | 当前品类的 `slug` · `template_path` · D10 档位 |
| 2 | `开发文档/模板引擎/品类核心参数规格_v1.0.md` | 该品类 **core / tuning / theme** 参数表 |
| 3 | `03-背景与调研/品类调研/{genre}/设计规范与实现路径.md` | 手感、单局时长、Godot 实现建议 |
| 4 | `config/optional_skills.json` | 该品类预制技能（S7 最多选 2 个） |
| 5 | `config/play_variants.json` | 有无玩法子模式 / 跨品类跳转 |
| 6 | `assets/ASSET_GAP_ANALYSIS.md` | 素材够不够用；不够则先补 P0 或写降级方案 |
| 7 | `templates/_examples/{genre}_game_config.json` | 若存在则作 JSON 结构参考 |
| 8 | `templates/platformer/` | **已有样板**：目录怎么分、config 怎么写 |

**产出（自检）**

- [ ] 能口头说清：玩家目标、输赢条件、几种操作
- [ ] 列出本品类 **core 锁定项**（Agent 将来不能改）
- [ ] 列出本品类 **tuning 可改项**（AI 创作只改这些）
- [ ] 确认素材包路径（或标记「需先下载 P0」）

**补充阅读**

**产出**

### ③ 细化完善要求 · 更新文档（约 20–40 分钟）

**目标**：开发前把歧义写死，避免工程做到一半改需求。

**要更新的文档**

| 文档 | 更新内容 |
|------|----------|
| `03-背景与调研/品类调研/{genre}/设计规范与实现路径.md` | 补充：L0 最小玩法范围、关卡/波次结构、节点预算 |
| `templates/_examples/{genre}_game_config.json` | 若无则新建；字段与 platformer 对齐 |
| `assets/ASSET_GAP_ANALYSIS.md` | 仅当本品类素材有新缺口/已补包时更新 |
| `assets/theme_paths.json` | 追加本品类默认 sprite 路径（若文件不存在则创建） |

**L0 最小范围模板（写入设计规范）**

```markdown
## L0 最小可玩范围（D10）
- 玩家目标：（例：收集 N 个 / 撑 N 波 / 到达终点）

- 操作键：≤3
- 必要场景：main · level/wave · player · （enemy/tower/bullet…）

```

**人工审理点（③ 出口）**

负责人扫一眼设计规范 §L0 范围：**同意** → 进入 ④；**驳回** → 改文档回到 ③。

### ④ 正式工程搭建（约 2–4 小时 / 品类）

**目标**：在 `templates/{genre}/` 下落完整 Godot 工程。

**标准目录（与 platformer 一致）**

```text
templates/{genre}/
├── project.godot
├── config/
│   └── game_config.json       ← AI 将来主要改这个
├── core/                      ← 🔒 预制逻辑，强类型 GDScript
│   ├── game_config_loader.gd  ← Autoload：GameConfig
│   ├── game_manager.gd
│   ├── …（移动/命中/波次等）
│   └── skills/                ← optional_skills 对应脚本
├── scenes/
│   ├── main.tscn
│   └── …
└── assets/kenney/…            ← Junction 指向全局 assets（不复制大文件）
```

**搭建顺序建议**

1. `project.godot` + Autoload `GameConfig`
2. `config/game_config.json` 默认值（tuning 在规格 ±30% 内）
3. `core/` 核心脚本（先跑通再 polish）
4. `scenes/` 场景（优先脚本生成平台/波次，控制节点数）
5. `assets/kenney/` Junction 到已有 Kenney 包
6. 技能：若 `optional_skills.json` 有该项 → `core/skills/{id}.gd` + 在 player/主控中挂钩

**禁止**

- 修改仓库根目录其他品类的 `templates/`
- 修改全局 `core/`（若存在）
- 引入非 `assets/` 素材

**platformer 样板对照**

| 模块 | platformer 实现 | 新品类应复用 |
|------|-----------------|--------------|
| 配置加载 | `core/game_config_loader.gd` | Autoload 模式 |
| 主控 | `core/game_manager.gd` | 读 config · 加载关卡 · HUD |
| 玩家 | `core/player_platformer.gd` | 改成该品类控制器 |
| 关卡 | `core/level_01.gd` | 程序化生成平台/波次，控节点数 |
| 验收 config | `config/game_config.json` | meta / tuning / theme / content_safety |

### ⑤ Godot 测试与打分（约 30–45 分钟）

**必做 MCP 流程**

```text
1. get_godot_version  → 4.6.3.stable
2. run_project(projectPath=templates/{genre})
3. get_debug_output     → errors 必须为空
4. stop_project
5. 人工试玩 2–3 分钟（移动/目标/输赢）
```

**L0 打分表（满分 100，≥80 且硬门槛全过 = 可进 ⑦）**

| 维度 | 分值 | 评分标准 |
|------|------|----------|
| **A. 可运行** | 25 | MCP 无 ERROR（25）；有 ERROR（0，硬 FAIL） |
| **B. 玩法闭环** | 20 | 有明确目标+输赢（20）；能玩但无结束（10）；无法玩（0） |
| **C. 配置驱动** | 15 | 改 tuning/theme 即可换数值/皮（15）；写死在场景里（5） |
| **D. 规格符合** | 15 | core/tuning 与品类规格一致（15）；缺项每项扣 3 |
| **E. K12 与安全** | 10 | 无违规内容+≤3 键（10）；超键数/内容风险（0） |
| **F. 工程约束** | 10 | 节点≤80、深度≤5、强类型（10）；超标每项扣 3 |
| **G. 展陈观感** | 5 | 有背景/UI/反馈（5）；纯色块（2） |

**打分记录模板**（复制到会话或 `开发文档/模板引擎/评审记录/{genre}_L0.md`）

```markdown
# {genre} L0 评审记录
- 日期：
- 评审轮次：R1 / R2 / R3
- MCP errors：有 / 无
- 总分：__/100
- 硬门槛：H1–H6 全 PASS / FAIL（项：）
- 扣分说明：
- 结论：通过 / 小循环 / 阻塞
```

### ⑥ 优化调整 · 小循环（每轮 ≤1 小时，最多 2 轮）

**触发条件**

- 总分 < 80
- 或任一硬门槛 FAIL
- 或人工试玩「手感明显不对 / 小孩玩不懂」

**小循环步骤**

```text
1. 根据打分表扣分项列 fix 清单（优先 H1→B→D）
2. 只改 templates/{genre}/ 内文件
3. 重新执行 ⑤ 全流程打分
4. 若第 2 轮仍 < 80 → 标记 blocked，人工决策砍 scope 或延期
```

**常见 fix 对照**

| 现象 | 优先改 |
|------|--------|
| MCP ERROR | 脚本类型/Autoload/场景路径 |
| 玩法无终点 | game_manager 加 win 条件 |
| 配置不生效 | GameConfig 字段名与 JSON 对齐 |
| 节点超标 | 改 level 脚本合并平台/敌人 |
| 素材缺失 | Junction 路径或 theme_paths 修正 |
