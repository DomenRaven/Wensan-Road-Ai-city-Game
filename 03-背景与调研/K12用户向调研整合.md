## 一、调研摘要（给用户向决策用）

本次补调研聚焦 **「谁在用、在什么场景、真正要什么、痛在哪里」**，补齐 Survey-01 的技术向缺口。

**对 GameForge K12 的直接结论：**

1. **时长对标**：行业 K12 互动多为 **40–60 分钟/站**（Scratch 课 40min、AI 乐园 60min、腾讯 AI 编程第一课 ~60min）；本项目 **8–10 分钟** 属于 **极短循环**，必须 **低门槛 + 强 preset + 可见进度**，不能照搬课堂/体验馆节奏。
2. **用户要的是「我做出了」而非「我学过了」**：少儿编程调研一致指出——**看作品不看知识点**；家长/教师担心 **模仿≠创造**；展陈应强调 **可试玩、可带走、可上墙**。
3. **亲子共学是默认模式**：6–8 岁需家长陪同熟悉操作；展陈应 **≤3 键**、**讲解员 30 秒复位**，避免「孩子卡住、家长代劳」变成黑盒。
4. **博物馆 UX 铁律**：**低门槛进入、短循环、并行工位、可旁观学习**；高放弃点 = **指令难懂 / 任务过难**；需记录 **drop-off 点** 与 **复位 SOP**。
5. **AI 认知定位**：2024–2025 国内 AI 教育从「技术演示」转向 **「人机协作创作闭环」**（腾讯 AI 编程第一课、梁溪 AI 体验馆）；与本项目 **「AI 创作透明厨房」** 一致，但须 **可视化改 config**，避免 ChatGPT 式黑盒。
6. **问题定义应收窄为**：不是「如何做全功能 AI 游戏引擎」，而是 **「如何在 K12 展陈 10 分钟内，让亲子完成一次可见的创作闭环」**。

## 二、目标用户与移情图

### 2.1 四类核心用户

| 用户 | 场景 | 表面需求 | 深层需求（移情） |
|------|------|----------|------------------|
| **6–12 岁孩子** | 散客/研学 | 好玩、快、能炫耀 | **「这是我做的」** 的身份认同；不想看 long loading |
| **陪同家长** | 亲子 | 孩子学东西、别白来 | 看见 **过程** 而非结果；可拍照/扫码；**别让我代操作** |
| **研学老师** | 20–40 人团 | 省心、有成果、可控时 | **批量流转**、**成果可展示**、**不出教学事故**（暴力/失败尴尬） |
| **现场讲解员** | 8h 值守 | 少出事、快复位 | **一键回 preset**、**双工位不堵**、**30 秒内下一组能上** |

### 2.2 移情图（亲子家庭 · 合成）

| 维度 | 内容 |
|------|------|
| **说** | 「妈妈我想做太空游戏」「这个要等多久？」「我能带回家吗？」 |
| **想** | 会不会很难？会不会 boring？别的小朋友会不会笑我？ |
| **听** | 讲解员介绍「10 分钟做一个小游戏」；旁边试玩屏的音效 |
| **关注** | 屏幕上的角色是不是 **我的主题**；AI 是不是真的在「帮我做」 |
| **痛点** | 等待生成、看不懂代码、操作太多键、失败白等、不能分享 |
| **收获** | 试玩到 **自己的游戏**、作品上墙/扫码、理解「AI 是工具不是代写」 |

### 2.3 移情图（讲解员 · 合成）

| 维度 | 内容 |
|------|------|
| **痛点** | 上一组改乱 config、生成失败占屏、排队积压、家长代操作引发纠纷 |
| **收获** | 流畅接待、孩子惊喜反馈、领导检查不翻车 |
| **关键需求** | **demo_preset 硬回退**、**复位 ≤30s**、**过程屏可读**（非黑客终端） |

## 三、POV 与 Problem Statement

### 3.1 POV（观点陈述）

> **K12 亲子家庭** 在 AI 教育区展陈场景下，需要 **在单次到访的有限注意力窗口内** 完成一次 **可见、可试玩、可带走证据** 的创作体验，因为 **孩子要的是「我创造了游戏」的身份感，家长要的是「AI 被看见地协作」而非黑盒代劳**；现有设备陈列/纯聊天 AI **无法满足** 这种 **短循环创造闭环**。

### 3.2 Problem Statement（问题阐述）

| 要素 | 定义 |
|------|------|
| **目标用户** | 6–15 岁 K12 学生 + 陪同家长；次要：研学团（老师带队） |
| **场景** | 文三路 AI 教育区 **3.0m×2.5m 体验岛**；单人 8–10min 或双工位并行 |
| **刚需** | 选主题 → **看见** AI 改配置 → **60 秒内** 可试玩 → 失败可 **立即** 回退 demo |
| **设计阈值** | 操作 ≤3 键；无暴力血腥；生成 **P90≤10min**（未实测前对外称「目标 P90」） |
| **非目标** | 不做 3A 品质；不做开放 ChatGPT；不要求孩子读懂全部代码 |

### 3.3 HMW（How Might We）· 3 条

1. **HMW** 让 10 岁孩子在 **不读代码** 的情况下，仍感到 **「这是我和 AI 一起做的」**？  
2. **HMW** 在 AI 生成失败时，让讲解员 **30 秒内** 恢复下一组可用状态？  
3. **HMW** 让研学团 **25 人/40 分钟** 内 **≥80%** 孩子拥有可展示成果（上墙/扫码）？

## 四、分主题调研要点

### 4.1 K12 用户画像与亲子研学

**行业共识：**

- 学校 AI 科技馆分级：**小学 40–60min 趣味体验**，高中 90–120min 项目创作（学校 AI 智慧科技馆方案）。
- 研学展厅情绪曲线：**好奇 → 认知 → 体验 → 升华**；强调 **参与感、仪式感、数字留存**（动植物保护研学展厅案例）。
- 高雄科工馆「职探任意门」：**亲子共游、在做中学**；多场景 **低文本、游戏化闯关**（KingOne 策展）。

**对本项目：**

- 用户矩阵（亲子/研学/打卡）与调研 **一致**；需补 **现场观察** 验证优先级。
- **「能出片」** = 过程可视化 + 试玩屏录 + 成果上墙——应作为 **发布层（成果发布）** 硬需求。

### 4.2 博物馆 儿童互动 UX（国际文献）

**FUTURE WORLDS（NCSU · 9–12 岁）设计原则摘录：**

| 原则 | 含义 | 本项目映射 |
|------|------|------------|
| Low barrier to entry | 短触达也可玩 | 首屏 ≤2 步进入试玩 |
| Collaboration | 多人/旁观 | 双工位 + 「看即学」 |
| Novel but accessible | 新但不难 | 预设模板 + 换皮 |
| Short OR long sessions | 长短皆可 | **8–10min 短循环** 是子集 |

**Keeper of Paintings（英国国家美术馆 · 7–11 岁）：**

- 数字体验 **是手段不是目的**；定期 **放下设备** 看实物（Skill Cards）。
- **启示**：试玩后应引导孩子 **讲给家长听**，而非一直盯着生成屏。

**Utsubo / 展陈运营（2026）：**

- 高放弃点 = **指令 confusing** 或 **任务 too hard**。
- 需测：**drop-off by segment**、**completion rate**、**queue throughput**。
- 个性化/创作类展项：**队列 + 组间 reset/cleaning** 是常态成本。

### 4.3 AI 教育体验馆对标

| 对标 | 时长 | 模式 | 启示 |
|------|------|------|------|
| 澳门 **AI 乐园** | **60min/票** | 7 站闯关：象棋/AI 艺术/VR/Alpha Mini… | 单站短、总站长；我们 = **单站浓缩** |
| 腾讯 **AI 编程第一课** | **~60min** 故事线 | 线稿成画、AI 作曲；**降低 Prompt 门槛** | 应用 **线稿/选主题** 而非纯文字 |
| 梁溪 **青少年 AI 体验馆** | 学校预约 + 周末散客 | 五大区、协作机器人、创新工坊 | **成果可见**（绘本、编程） |
| 新加坡 **艺术科学博物馆** 趣味编程 | **60min** 工作坊 | 20–25 人/场；Lego 编程 | 团次规模参考 |
| Scratch Jr 线上课 | **40min/次** | 听故事→做作品→秀作品 | **40min** 是儿童 **专注上限**参考 |

**关键洞察：** 几乎没有 **10 分钟「从零到可玩小游戏」** 的成熟展陈对标；本项目是 **差异化卖点**，也是 **风险点**——必须靠 **core 预制** 而非现场开发。

### 4.4 少儿游戏创作 · 用户痛点

| 来源 | 痛点 | 设计响应 |
|------|------|----------|
| 新华网（少儿编程调查） | 机构 **慢且贵**；Scratch **可自学**；低龄 **模仿≠理解** | 不做「教语法」；做 **换主题即成品** |
| kidsaitools（2026） | **3 节课做完整游戏** = 老师代做 | 禁止 hidden 代写；过程屏可见 |
| 天工 AI 课设文章 | **40min** = 5 导入 + 25 做 + 7 秀 + 3 结 | 映射：**1min 选主题 + 5min 生成 + 3min 试玩** |
| 童程 Scratch Jr | 前 **3 次** 要家长陪 | UI **自解释**；讲解员 **15 秒指引** |

### 4.5 展陈运营 · 讲解员 SOP（文献合成）

**每日 / 每组：**

1. **开机自检**：MCP run demo_preset **无 ERROR**（<60s）  
2. **组间复位**：Session 结束 → **一键 preset** → 清 workspace  
3. **失败降级**：生成超时 → **自动 demo_preset** + 文案「先玩这款，后台继续」  
4. **日志**：记录 **放弃步骤**（选题/生成/导出/试玩）  
5. **graceful degradation**（blooloop 巡展）：关键路径失败时 **预定义访客体验**，员工 **有权 reboot** 而无需等开发  

**双工位：**

- 并行站缩短 **有效排队**；需 **清晰起止区**（「观看即学会」）。  
- 峰值 **0.6kW** 下避免 **双 Session 同时导出**——调研建议 **错峰导出** 或 **先玩 exe 后导出**。

### 4.6 创作透明化 · AI 搭档认知

| 趋势 | 证据 | 本项目 |
|------|------|--------|
| 从演示到协作 | 腾讯 AI 第一课、香港 AI 亲子工程师（Gemini 反馈循环） | 过程屏 = **透明厨房** |

| 科技/设计向善 | 课程大纲、2026 大作业 AIGC 讲故事 | 无暴力、≤3 键、可讲解 |

## 五、用户旅程与放弃点（问题定义 · 旅程分析）

```
到达 → 听讲解(30s) → 选主题(60s) → 看AI改config(3-5min) → 生成/运行(1-3min) → 试玩(2-3min) → 成果/扫码(1min) → 离开
         │              │                │                      │
         放弃A          放弃B            放弃C(难懂/无聊)      放弃D(失败/白屏)
```

| 放弃点 | 可能根因 | 对策 |
|--------|----------|------|
| A | 不知道怎么开始 | 讲解员脚本 + 屏幕 **idle 演示** |
| B | 主题太多 | **≤6 个主题 preset**，大图卡片 |
| C | 过程像黑盒 | **可视化 diff / 阶段条** |
| D | 生成失败 | **demo_preset 硬回退** + 计时 SLA |

## 八、对 v1.1 方案的用户向修正建议

1. **对外话术**：「**一次到访 10 分钟内试玩自己的主题游戏（目标 P90）**」，不说「学会游戏开发」。  
2. **首屏 UX**：**1 键选主题** + idle 循环 demo（降低放弃 A/B）。  
3. **过程屏**：至少 **4 阶段**（理解需求→改配置→运行→试玩），每阶段 **≤15 字** 儿童可读文案。  
4. **研学模式**：预置 **班级主题包**（老师选 1 个），避免 25 个孩子 25 个主题。  
5. **成果发布**：试玩结束 **自动 5 秒高光 replay** + 二维码（满足家长「出片」）。  
6. **讲解员手册**（Wave 1 必写）：复位、降级、双工位错峰、**禁止词**（不说「AI 帮你写好了」→ 说「AI 根据你的主题调整了游戏」）。
