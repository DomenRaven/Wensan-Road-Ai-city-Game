## 一、产品定位（与你方差异的起点）

| 维度 | 百度秒哒 | GameForge K12（本项目） |
|------|----------|-------------------------|
| **目标用户** | 87% 非程序员；OPC、教师、小微企业主 | K12 展厅；14 岁以下；讲解员引导 |
| **产物形态** | 网页 / 小程序 / H5 / 原生 APP / 轻小游戏 | Godot 4.6 可执行游戏（Windows exe） |
| **技术路线** | 文心大模型 + **全栈代码生成**（React/Taro 等） | **预制 L0 模板** + 仅改 `game_config.json` |
| **生成深度** | 前端 + 后端 + DB + 部署 + 商业化插件 | 玩法/主题/数值 tuning；`core/` 锁定 |
| **典型耗时** | 宣称 3–5 分钟首版，1 小时迭代 | 10 分钟向导 + Agent 写 config + 试玩 |
| **成熟度感知** | 20 万+ 应用、应用广场、企业版 SLA | 11/11 L0 ✅；Kiosk 🔄；人工审核进行中 |

**结论**：秒哒成熟度高，主要来自 **「端到端全栈交付 + 多 Agent 编排 + 云托管闭环」**，而非单点「游戏引擎精度」。你方若在「小游戏手感」上对标，需区分：**秒哒的小游戏多为 H5/互动页**，不是 Godot 级原生游戏。

## 二、技术架构（三层 + 多智能体）

### 2.1 官方表述的三柱

```
┌─────────────────────────────────────────────────────────────┐
│  用户层：LUI 自然语言  +  GUI 拖拽编辑  +  多模态输入(图/文档) │
├─────────────────────────────────────────────────────────────┤
│  编排层：中枢调度引擎 → 10+ 专业智能体（A2A 协同）            │
│         策划 | 开发 | 设计 | 文案 | 测试 | 安全 | 架构师 …    │
├─────────────────────────────────────────────────────────────┤
│  能力层：MCP 统一工具调用（地图/语音/支付/可灵生图/搜索…）     │
│         文心大模型 · RAG · Context Linking · 数据飞轮        │
├─────────────────────────────────────────────────────────────┤
│  运行时：百度智能云一键部署 · Supabase 后端 · CDN/域名/小程序  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 两大协作协议（官方技术披露）

| 协议 | 作用 | 公开指标 |
|------|------|----------|
| **A2A**（Agent-to-Agent） | 智能体间消息传递、任务分派、交付物整合 | 协同误差率 < 0.5% |
| **MCP**（Model Context Protocol） | 统一外部工具/API 调用接口 | 上百种云能力与第三方插件 |

### 2.3 上下文与质量保障

- **Context Linking**：修改局部逻辑时自动关联历史模块（如改支付 → 联动订单/统计），准确率宣称 > 98%
- **进化式工作流**：用户行为反馈进入训练闭环（如多次要求「登录」→ 后续默认生成登录模块）
- **算力调度**：万级并发下单应用生成稳定在 ~3 分钟内；峰值支持 10 万级并发操作

### 2.4 导出技术栈（源码可见部分）

官方「源码导出」文档披露生成物典型栈：

| 层级 | 技术 |
|------|------|
| 前端 | **React**；小程序侧 **Taro**（`npx taro build --type h5/weapp`） |
| 后端/存储 | **Supabase**（需自行配置 URL/Anon Key；`VITE_SUPABASE_PROXY=false`） |
| 构建 | Node.js + npm + Vite |
| 三方 API | 通过 Vite proxy 或替换 `utils/api.ts` 基址 |

> 秒哒 **不生成 Godot 项目**；小游戏多为 Canvas/DOM/React 实现的 H5 互动或视觉小说。

## 三、用户工作流（官方五步法）

来源：[快速入门](https://cloud.baidu.com/doc/MIAODA/s/Dm8ecpgsg)

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 描述需求 | 自然语言 / 上传需求文档 / 截图草图 | 可选「指令优化」扩写 Prompt |
| 2 需求澄清 | 多轮对话补全（名称、风格、功能清单） | 产品经理智能体预判常规模块（登录、导航等） |
| 3 生成应用 | 确认后等待 3–5 分钟 | 多智能体并行：架构 → 开发 → 设计 → 测试整合 |
| 4 修改应用 | LUI 对话改细节 + GUI 可视化编辑 | 局部修改走 Context Linking，避免破坏全局 |
| 5 发布分享 | 一键发布百度云 / 小程序 / 自定义域名 | 自动配置服务器与 DB，无需用户运维 |

### 3.1 交互设计原则（产品负责人原话）

- **首次创建以 LUI 为主**，后续迭代 **LUI + GUI 双轨**
- 代码与基础设施 **对用户不可见**（所见即所得预览）
- **应用广场**：公开需求对话过程 + 成品，支持克隆二创

### 3.2 「可落地投产」方法论（官方最佳实践）

来源：[如何做可落地投产的应用](https://cloud.baidu.com/doc/MIAODA/s/Zmkytqytk)

1. **清晰需求**：类型、页面结构、集成能力、设计风格五要素写全  
2. **小步快跑**：MVP → 多轮叠加；用 Mermaid 流程图描述交互  
3. **提问式生成**：不确定时让模型给多方案再选  
4. **美观专项 Prompt**：单独一轮做视觉风格升级

## 五、小游戏 / 游戏类能力（重点对标）

### 5.1 秒哒「游戏」实际是什么

| 类型 | 实现方式 | 官方案例 |
|------|----------|----------|
| 休闲 H5 小游戏 | 生成 Web 交互（如 Flappy 类） | InfoQ：「飞翔的小鸟」一句话生成 |
| 剧情向视觉小说 | React DOM + 节点图 + 可灵生图 | [剧情向小游戏搭建指南](https://cloud.baidu.com/doc/MIAODA/s/Pmpz55jdh) |
| 模板化小游戏 | 消除/跑酷/答题等模块化模板 | 百度智能云营销文：三步选模板→自定义→发布 |

### 5.2 官方推荐的「剧情游戏」三步工作流

```
第一步 搭骨架  →  一条结构化 Prompt → MVP（纯文字+分支交互）
第二步 写好戏  →  编剧准则 Prompt → ~10 分钟完整剧本
第三步 做画面  →  @可灵技能 → 场景图 + 人物立绘自动映射
```

**技术细节（文档级）**：
- UI：全屏插图 + 底部字幕层（≤25% 屏高）；**DOM 渲染文本，禁止 Canvas 打字机**
- 玩法：点击推进 / 左右头像对话 / 分支 / 多结局
- 已知坑：React 打字机组件需 `key` 重置，官方提供修复 Prompt

### 5.3 与 Godot 路线的本质差距

| 能力 | 秒哒 | GameForge K12 |
|------|------|---------------|
| 物理/碰撞/帧同步 | 弱或没有（Web 逻辑） | `core/` 预制 |
| 手感调参（跳跃、射击反馈） | 依赖生成代码质量，无统一 schema | `tuning` ±30% 有规格约束 |
| 素材一致性 | AI 生图 + 通用 UI | Kenney CC0 + theme_paths |
| 离线 exe / 展厅 Godot 试玩 | 非主路径 | 主路径 |
| 可预测性 | 每次生成代码不同 | 模板 deterministic |

## 八、为什么秒哒「感觉更成熟」— 可拆解的 8 条

1. **端到端交付**：不只生成草稿，而是可访问 URL + 后台管理 + 数据持久化  
2. **多 Agent 显式分工**：用户感知到「团队在干活」，降低不信任  
3. **双模编辑**：聊完还能拖拽改，弥补纯 LLM 不稳定  
4. **需求澄清产品化**：指令优化、多轮追问、广场抄作业  
5. **工具/插件市场**：支付、登录、地图等一次对话接入  
6. **发布零运维**：百度云自动部署，用户无需懂 Redis/Godot/export  
7. **数据飞轮 + 规模**：20 万应用反馈迭代；你方仍处 L0 人工审核  
8. **文档与 SOP 极细**：如剧情游戏三步、打字机 bug 修复 Prompt、Mermaid 流程图建议

## 九、对 GameForge K12 的可借鉴项（按优先级）

### P0 · 工作流感知

- [ ] **★R 配方页**做成「虚拟团队交付清单」（策划/美术/调参/测试）— 对齐秒哒多 Agent 可见性  
- [ ] **分步确认**：需求摘要 → 用户点头 → 再生成（秒哒第二步「需求澄清」）  
- [ ] **小步迭代 Prompt 规范**：禁止一次改完全部 tuning；MVP config → 多轮微调  

### P1 · 完成度闭环

- [ ] **试玩即 URL/exe**：S9 一键启动无 ERROR（对标秒哒「3 分钟可玩」）  
- [ ] **生成过程可视化**：展示当前 Agent/步骤（哪怕模拟）  
- [ ] **降级链 demo_preset**（文档 #18）— 秒哒广场克隆思路  

### P2 · 游戏专项

- [ ] **剧情类**：可参考秒哒「骨架→剧本→美术」三阶段，映射为 `theme` → `narrative` → `assets`  
- [ ] **质检清单自动化**：对标秒哒 AI 质检 + 你方 `l0_manual_audit.py`  
- [ ] 明确对外话术：**「Godot 原生手感」vs「H5 轻互动」** 差异化  

### 不建议照搬

- 全栈代码生成（违背 `core/` 锁定与 K12 可控性）  
- Supabase/React 栈（与 Godot 展厅路线冲突）  
- 无限制 Agent 改代码（秒哒可维护性差，你方模板化是优势）
